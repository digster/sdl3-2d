# 2026-05-21 — fifth example: SDL3 + bgfx (alternate GPU abstraction)

Added a fifth, fully independent example: `examples/cpp-bgfx/`, the same
hello-triangle as `cpp-gpu/triangle.cpp` but rendered through
[bgfx](https://github.com/bkaradzic/bgfx) instead of SDL_GPU. Two binaries
from one folder so the SDL3↔bgfx wiring and the GPU pipeline are taught
separately:

- `demo_bgfx` (`hello_bgfx.cpp`) — clear + bgfx debug-text overlay. No
  shaders. ~225 lines. Teaches the genuinely new piece: passing the native
  `NSWindow*` from SDL3 into `bgfx::PlatformData`, the
  `bgfx::renderFrame()`-before-`init()` single-threaded-mode opt-in, and
  the `setViewClear` / `touch` / `frame` per-frame lifecycle.
- `demo_bgfx_triangle` (`triangle_bgfx.cpp`) — same wiring + a full
  hello-triangle with `bgfx::VertexLayout`, vertex/index buffers via
  `bgfx::makeRef`, and offline-compiled `.sc` shaders loaded as `.bin`
  bytecode at runtime via `bgfx::createShader` / `bgfx::createProgram`.
  ~350 lines.

## Files added

- `examples/cpp-bgfx/hello_bgfx.cpp` — wiring demo, narrated block-comment
  header explaining the three pieces of SDL3↔bgfx wiring + the bgfx

- `examples/cpp-bgfx/triangle_bgfx.cpp` — triangle demo. First half is

## Key decisions

1. **Independent example folder, no shared source.** Same independence rule
   as the four pre-existing examples — `cpp-bgfx/` builds with only its own
   folder on the include path. Both `.cpp` files re-declare their own
   `get_native_window_handle()` helper, AppState struct, and `fail()`
   helper rather than reach into each other. The duplication is the cost
   of independence (same trade-off as `c/gfx.c` vs `cpp/gfx.cpp`).

2. **Two binaries from one folder, not one binary with a `--mode` flag.**
   Considered the shadercross-style `--mode hello|triangle` pattern but
   rejected it: the wiring demo's whole point is that it has *no shader
   pipeline at all*. Splitting cleanly between `hello_bgfx.cpp` (zero
   bgfx-side machinery beyond the wiring) and `triangle_bgfx.cpp` (full
   pipeline) makes the additions on top of the wiring readable as a diff,
   which is what teaches.

3. **Document-only install, mirrors shadercross 1:1.** bgfx is not in
   Homebrew. Both build systems probe `BGFX_PREFIX` and skip the bgfx
   targets cleanly when it is absent. The "just `brew install sdl3`"
   promise still holds for the three core demos (`c/`, `cpp/`, `cpp-gpu/`),
   and the bgfx detection is independent of the shadercross detection so
   either optional can be installed without the other.

4. **Two-shape detection (installed prefix preferred, in-tree-build fallback).**
   `bgfx.cmake` produces an unusual in-tree layout when used without
   `cmake --install` — headers in `bgfx/include/`, libs in
   `cmake-build/cmake/bgfx/`. The Makefile and CMake both probe the
   canonical installed shape first (`$PFX/include/bgfx/bgfx.h` +
   `$PFX/lib/cmake/bgfx/bgfxConfig.cmake`) and fall back to the in-tree
   shape, so users who skip `cmake --install` still get a working build.

5. **`bgfx.cmake`'s install rules don't ship `bgfx_shader.sh` /
   `bgfx_compute.sh`** — an upstream omission. Discovered during the
   "compile a probe `.sc`" verification: shaderc errored on
   `#include <bgfx_shader.sh>` because the install prefix's `share/bgfx/`
   only contained licenses. **Fix**: a `cp` step (d) in the install
   instructions copies the two `.sh` headers from the source tree into
   `$(BGFX_PREFIX)/share/bgfx/shaders/`. Document this in both the
   `cpp-bgfx/README.md` install sequence and the `ARCHITECTURE.md`
   conventions list. The Makefile's `BGFX_SHADER_INC` detection probes
   the canonical share/ path first, then the sibling source tree as a
   fallback, so the example builds either way.

6. **`bgfx::renderFrame()` MUST be called once BEFORE `bgfx::init()`.**
   Documented opt-in to bgfx's single-threaded mode. Without it bgfx
   spawns a render thread that fights SDL3's main-thread expectation on
   macOS, complicates shutdown ordering, and adds a noticeable stall as
   bgfx joins its worker on `bgfx::shutdown()`. Both demos do this; the
   ARCHITECTURE.md conventions list now documents it.

7. **bgfx's `varying.def.sc` parser silently drops declarations that
   follow `//` line comments.** Discovered the hard way: had narrative
   comments above each declaration in `varying.def.sc`, shaderc's
   resulting HLSL intermediate dropped `a_position` from the
   function signature, fragment of debugging surfaced the parse quirk
   only after `--preprocess` dumped the generated HLSL. The fix is to
   keep `varying.def.sc` mechanical — declarations only, blank lines
   fine, no `//` comments. Narrative comments now live in the `.sc`
   shader files (which use a real C preprocessor) and the sibling
   `shaders/README.md`. Documented in ARCHITECTURE.md.

8. **Metal-only shader compilation by default, with documented
   cross-platform reflag.** Same scope decision as `cpp-gpu/triangle.cpp`
   (embedded MSL, macOS-only): keep the example focused on the wiring
   rather than the build matrix. `examples/cpp-bgfx/shaders/README.md`
   documents the `shaderc -p spirv` / `-p s_5_0` re-runs for other
   backends; the host C++ would extend `load_shader_blob()` to pick the
   right subdirectory based on `bgfx::getRendererType()`.

9. **`bgfx::frame()` always returns whether a display is attached or not.**
   Unlike the SDL_GPU examples, the bgfx demos don't need NULL-swapchain
   handling for headless smoke runs — `--frames N` exits 0 even when no
   display is present.

## Verification

End-to-end checks all passed:

- `make smoke` runs all five demos (six binaries counting the bgfx pair)
  for 120 frames headless, exits 0. Both bgfx demos log
  "bgfx hello / triangle: renderer = Metal" so smoke output proves they
  ran (added during debugging — bgfx's debug-text overlay doesn't go to
  stdout).
- CMake path works: `cmake -B build && cmake --build build` builds both
  bgfx demos through `find_package(bgfx CONFIG)` against the installed
  prefix.
- Missing-shaders error path: removing
  `examples/cpp-bgfx/shaders/build/metal/vs_triangle.bin` and running
  `./build/demo_bgfx_triangle` produces a clear "Run `make bgfx-shaders`
  first" message and exits 1.
- Isolation check: `cd examples/cpp-bgfx && clang++ -std=c++17 -I.
  -I$PFX/include hello_bgfx.cpp $(pkg-config --cflags --libs sdl3)
  -L$PFX/lib -lbgfx -lbimg -lbx -framework Metal …` builds standalone
  and runs cleanly with `--frames 30`.

# 2026-05-19 — Split into two fully independent C / C++ examples

## What was done

Reversed the original "one shared C `gfx.o` linked by both demos via
`extern \"C\"`" design into two standalone, parallel examples that share no
source.

New layout:

```
examples/
  c/    gfx.h   gfx.c   traditional.c   -> build/demo_traditional   (C)
  cpp/  gfx.hpp gfx.cpp callbacks.cpp   -> build/demo_callbacks      (C++)
Makefile        (top-level, 2 independent targets, no shared gfx.o)
CMakeLists.txt  (top-level, 2 independent executables, no shared gfx lib)
```

- `examples/c/gfx.h` — old `include/gfx.h` with the `#ifdef __cplusplus` /
  `extern "C"` guard removed (dead now; only C consumes it) and the
  interop/name-mangling narrative rewritten.
- `examples/c/gfx.c`, `examples/c/traditional.c` — moved from `src/` /
  `examples/`, comments updated for the new paths/independence.
- `examples/cpp/gfx.hpp` + `gfx.cpp` — idiomatic C++ port: API in a
  `namespace gfx`, `gfx_` prefix dropped (`gfx::clear`, `gfx::fill_triangle`,
  …), no `extern "C"`, `std::sqrt`/`static_cast`/`nullptr`. Algorithms
  unchanged (midpoint circle, scanline fill, `SDL_RenderGeometry` triangle).
- `examples/cpp/callbacks.cpp` — moved; all `gfx_*` call sites rewritten to
  `gfx::*`; include + header framing de-interop'd.
- `Makefile` / `CMakeLists.txt` — rewritten: each demo built only from its own
  folder, private include path (`-Iexamples/c` / `-Iexamples/cpp`), separate
  object trees (`build/c`, `build/cpp`). No shared object/library. All
  existing targets/UX preserved (`all run-c run-cpp smoke debug-c debug-cpp
  clean check-sdl`, `DEBUG=1`).
- Deleted: `include/` (`gfx.h`), `src/` (`gfx.c`), flat `examples/*.{c,cpp}`.
- Docs: ARCHITECTURE.md rewritten (new "two standalone starters" thesis,
  component map with no cross-arrow, build table, conventions incl. "keep both
  gfx copies behaviourally identical — do not DRY back"). README.md table /
  API snippet (C `gfx_` vs C++ `gfx::`) / Customising note updated. PROMPT.md
  appended.

## Key decisions / context

- Per user: each example owns its gfx in its own language; one top-level build
  with independent targets; the interop demo is **replaced**, not kept.
- "Idiomatic C++" was interpreted as a `gfx` namespace with the prefix dropped
  — minimal, no new abstractions (no color struct / RAII), matching the
  "tiny, not a framework" intent.
- The ~135-line gfx duplication is intentional and documented as such, so
  either folder is copy-paste independent.

## Status / verification

- `make clean && make` — both demos build with `-Wall -Wextra -O2`, **zero
  warnings** (C as C11, C++ as C++17).
- `make smoke` — both run `--frames 120` non-interactively, exit 0.
- Independence proven: `cd examples/c && clang -I. *.c $(pkg-config --cflags
  --libs sdl3)` and the analogous C++ build each succeed with the *other*
  folder literally invisible.
- Remaining (user-only): interactive visual check via `make run-c` /
  `make run-cpp` (no behavioural change expected — same scene + bouncing ball,
  only the structure changed).

# 2026-05-19 (later) — Fix SDL3 IntelliSense errors in VS Code

## Problem

VS Code showed red squiggles + ~15 phantom errors (`Unknown type name
'SDL_Renderer' clang(unknown_typename)`) on every SDL3 file; build was fine.
Root cause: clangd parses files with no compiler flags, so `<SDL3/SDL.h>`
(`/opt/homebrew/include`, Apple Silicon Homebrew) doesn't resolve. Pure
editor-config issue, not a code issue.

## What was done

- `CMakeLists.txt` — added `set(CMAKE_EXPORT_COMPILE_COMMANDS ON)` after
  `project()` (with a why-comment).
- `Makefile` — new `compdb` target (added to `.PHONY` + targets doc):
  `cmake -S . -B build/cmake` then symlink `compile_commands.json` to root.
  `clean` now also `rm -f compile_commands.json` (kills the dangling symlink).
  Configured into `build/cmake/` so CMake never collides with the Makefile's
  `build/c` `build/cpp` object trees.
- `.vscode/` (committed — template correct on clone): `extensions.json`
  (recommends clangd), `settings.json` (clangd args +
  `--compile-commands-dir`, `--header-insertion=never`; Microsoft engine
  `disabled` to avoid duplicate squiggles, with documented toggle),
  `c_cpp_properties.json` (Microsoft-engine fallback pointed at the same DB).
- `.clangd.old` — inert zero-build fallback (`CompileFlags.Add:
  [-I/opt/homebrew/include]`); rename to `.clangd` to use without CMake.
- Docs: README.md ("Editor / IDE setup" section + `make compdb` in target
  list), ARCHITECTURE.md (editor DB as a read-only third consumer of the
  build), PROMPT.md appended.

## Key decisions / context

- User chose CMake compile DB as primary AND wanted the static `.clangd`
  shipped as `.clangd.old` (inert, documented, rename-to-activate).
- User unsure which extension → support both; default to clangd (its
  `clang(unknown_typename)` label was in the screenshot; `/usr/bin/clangd`
  present) with the Microsoft engine off; one-word toggle to swap.
- Symlink (not copy) so the regenerable machine-specific DB stays in
  `build/cmake/` and remains git-ignored (`.gitignore:19`); `.vscode/` and
  `.clangd.old` are tracked (verified via `git check-ignore`).

## Status / verification

- `make compdb` → root `compile_commands.json` symlink present; each entry
  carries its own `-Iexamples/{c,cpp}` + `-I/opt/homebrew/include` +
  `-std=gnu11`/`-std=gnu++17`.
- `make clean && make all` → fresh build of both demos, SDL3 flags intact,
  no warnings.
- `make smoke` → both demos ran `--frames 120`, exit 0 (config change does
  not affect the build).
- Remaining (user-only): reload VS Code / restart clangd and confirm the 15
  problems are gone in `examples/cpp/callbacks.cpp`.

# 2026-05-19 (later) — Add a third example: C++ SDL_GPU graphics pipeline

## What was done

Added a **third independent example** demonstrating SDL3's `SDL_GPU` API (the
abstraction over Metal/Vulkan/D3D12) — the canonical "hello triangle". The
repo's two prior examples both use the high-level 2D renderer; this one drops a
tier to show the explicit graphics-pipeline machinery.

New layout addition:

```
examples/cpp-gpu/  triangle.cpp  -> build/demo_gpu  (C++, SDL_GPU)
```

- `examples/cpp-gpu/triangle.cpp` — single self-contained TU, callback model
  (`SDL_MAIN_USE_CALLBACKS`, mirrors `callbacks.cpp`'s shape: heap `AppState`
  via `appstate`, same event handling). Lifecycle: create GPU device
  (`SDL_GPU_SHADERFORMAT_MSL`, debug on) → `SDL_CreateWindow` (NOT
  `…AndRenderer`) → `SDL_ClaimWindowForGPUDevice` → compile vertex+fragment
  shaders → build pipeline once → per frame: acquire cmd buffer → acquire
  swapchain → (if non-NULL) render pass + bind + `SDL_DrawGPUPrimitives(3,1,0,0)`
  → submit. Releases pipeline/window/device in `SDL_AppQuit`; does NOT call
  `SDL_Quit()` (callback runtime does).
- **Shaders embedded as MSL raw strings** (vertex baked, `[[vertex_id]]`, no
  vertex/transfer buffer) — Metal compiles at runtime, so zero new build deps
  (keeps the "just `brew install sdl3`" promise; SPIR-V would need shadercross).
- Build wiring: Makefile (`GPU_DIR`, `GPU_BIN`, `GPU_CXXFLAGS` with private
  `-Iexamples/cpp-gpu`, `build/cpp-gpu/` object tree, `run-gpu`, `debug-gpu`,
  added to `all`/`smoke`/`.PHONY`/header). CMake (`demo_gpu` target). `.gitignore`
  gained `demo_gpu`.
- Docs: README (three-example reframe, table row, targets, new "The GPU
  pipeline example" section, Customising notes), ARCHITECTURE (reframed
  "two→three, each isolates one axis"; 3-col component map; GPU render
  lifecycle; build/conventions/key-files updated incl. embedded-MSL +
  NULL-swapchain conventions), PROMPT appended, this memory entry.

## Key decisions / context

- Per user (planning): SDL callback loop (consistency with `cpp/`) + minimal
  static triangle (no vertex/transfer buffers) — tightest focus on the
  pipeline itself.
- Folder `cpp-gpu` (keeps language-prefix convention, signals the new axis);
  binary `demo_gpu`. No `gfx` — `SDL_GPU` and `SDL_Renderer` are mutually
  exclusive, so the example is one file and shares nothing.
- API verified against installed `/opt/homebrew/include/SDL3/SDL_gpu.h`
  (SDL3 3.4.8) before coding (signatures/struct fields/enum names).
- NULL swapchain handled as a valid no-draw frame (still submit) — this is
  what makes headless `make smoke` exit 0.

## Status / verification

- `make clean && make` — all **three** demos build, `-Wall -Wextra -O2`,
  C++17, **zero warnings**.
- `./build/demo_gpu --frames 60` and `make smoke` (all three `--frames 120`):
  exit 0. Metal backend confirmed selected ("Metal API Validation Enabled").
- Independence: `clang++ -std=c++17 -Iexamples/cpp-gpu
  examples/cpp-gpu/triangle.cpp $(pkg-config …)` builds standalone.
- Remaining (user-only): interactive visual check via `make run-gpu` — expect
  a window with one red/green/blue gradient triangle on a dark background;
  `Esc`/close quits cleanly. (Needs a display + Metal; can't be auto-verified.)

# 2026-05-19 (later) — Add `docs/` recipe site (24 SDL3 C++ recipes)

## What was done

Added a static HTML recipe site under `docs/` that doubles as a beginner →
advanced curriculum for the repo's two SDL3 styles. **28 new files**, no
changes to `examples/`, `Makefile`, or `CMakeLists.txt`.

```
docs/
├── index.html                       landing page, two-card hero (2D / 3D)
├── style.css                        shared sheet (Linear/Stripe-ish minimal,
│                                    custom-property tokens, badges, footer
│                                    nav, narrow-viewport tweaks)
├── 2d/
│   ├── index.html                   curriculum table-of-contents
│   ├── 01-hello-window.html         …through…
│   └── 12-mini-game-pong.html       (12 recipes, beginner → advanced)
└── 3d/
    ├── index.html
    ├── 01-hello-triangle.html       …through…
    └── 12-render-to-texture.html    (12 recipes)
```

Curriculum:

- **2D track** (uses `SDL_Renderer` + the existing `gfx::` helpers):
  hello window · drawing primitives · input · delta time · textures ·
  sprite animation · collision · audio · bitmap text · tilemap · particles ·
  Pong capstone.
- **3D track** (uses `SDL_GPU`, mirrors the patterns from
  `examples/cpp-gpu/triangle.cpp`): hello triangle · cross-platform shaders ·
  vertex buffers · index buffers · uniforms/MVP · 3D cube · depth testing ·
  textures &amp; samplers · Phong lighting · multiple objects · FPS camera ·
  render-to-texture capstone.

Every recipe HTML follows the same shape: breadcrumb top-nav, difficulty
badge, "What you'll learn", "Prerequisites", "Walkthrough" (narrative with
short snippets), "Complete listing" (drop-in replacement for the matching
`examples/cpp/callbacks.cpp` or `examples/cpp-gpu/triangle.cpp`), "Try it
yourself", prev/index/next footer nav.

All C++ snippets adhere to the codebase style: `SDL_MAIN_USE_CALLBACKS` +
the four `SDL_App*` functions, heap `AppState`, `true == success`, float
coords, no globals, no `using namespace std`. Procedural data (checker
textures, sprite sheets, bitmap font glyphs, cube geometry) is built in
code so each recipe is runnable without any asset files.

## Choices worth noting

- **Format**: static HTML with shared `style.css`, syntax highlighting via
  Prism.js from a CDN (`cdnjs` → `prism-tomorrow` theme + `prism-cpp`
  component). No build step, no JS frameworks; opens directly with `open
  docs/index.html`.
- **No extra buildable examples**: recipes are docs-only. The user's prior
  plan answer was "self-contained snippets in HTML" — confirmed not to
  touch the existing build.
- **README**: added a "Recipes (tutorial track)" section after "The GPU
  pipeline example" and before "Customising", per the global CLAUDE.md
  README-sync rule.

  recipes), plus `docs/style.css` = **28** files under `docs/`. README +
